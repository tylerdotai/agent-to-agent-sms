# Agent-to-Agent Verification Protocol

> How two agents establish trusted communication without a human in the loop.

---

## The Problem

Sendblue requires bidirectional contact verification before agents can SMS each other:

```
Agent A adds Agent B as contact → status: pending
Agent B must text Agent A to verify → status: verified
```

For **human-to-agent** this works fine — the human texts first. For **agent-to-agent**, neither agent can text first without already being verified. Circular dependency.

The solution: a **verification handshake** that establishes agent identity before contact verification is attempted.

---

## The Protocol: Shared Secret + Challenge-Response

Each agent pair uses:
1. A **shared secret** — exchanged out-of-band via a trusted channel (Telegram chat between the two humans)
2. A **keypair** — each agent has a Ed25519/ECDSA keypair for message signing
3. **Challenge-response** — proves both agents hold the shared secret without transmitting it

### Step 1: Human-Mediated Secret Exchange (One-Time Setup)

```
Human A (Tyler) ———— Telegram ——— Human B (Nikita)
"Here's my agent's secret: 7f3e9a2c..."
```

This is the only step requiring human involvement. The secret is a 32-byte random string generated by Agent A. The humans exchange it over Telegram (which is encrypted and authenticated).

**Agent A generates the secret:**
```bash
openssl rand -hex 32
# e.g.: 7f3e9a2c1d4b8f6e0a9c3d7e2b5f8a1c4d9e6f3b8a2c5d7e9f1b4a3c8d6e2f9b
```

Agent A stores this in its `verification.json` (see DATA_STRUCTURES.md).

**Agent B stores the same secret** in their copy, keyed to Agent A's ID.

---

### Step 2: Keypair Generation (Per-Agent, One-Time)

Each agent generates a signing keypair:

```bash
# Ed25519 example
openssl genpkey -algorithm ED25519 -out agent_private.pem
openssl pkey -in agent_private.pem -pubout -out agent_public.pem

# Or for ECDSA:
openssl ecparam -genkey -name prime256v1 -noout -out agent_private.pem
openssl ec -in agent_private.pem -pubout -out agent_public.pem
```

The **public key** is shared during the handshake. The private key never leaves the agent's host.

---

### Step 3: Agent Registry (agents.json)

Each agent publishes their public identity to `agents.json` in the repo:

```json
{
  "agents": {
    "nikita-agent": {
      "id": "nikita-agent",
      "name": "Nikita's Agent",
      "sendblue_number": "+1 (555) 234-5678",
      "public_key": "-----BEGIN PUBLIC KEY-----\nMFkwEwYHKoZIzj0CAQYIKoZIzj0DAQcDQgAE...\n-----END PUBLIC KEY-----",
      "telegram_chat_id": "123456789",
      "created_at": "2026-03-26T00:00:00Z"
    }
  }
}
```

This is the **address book** agents use to look each other up. Humans maintain this file via PRs.

---

### Step 4: Challenge-Response Handshake

Once agents have each other's public keys and shared secrets, they perform the handshake:

```
Agent A                                           Agent B
   |                                                  |
   |  1. A generates nonce_123 (random 16 bytes)     |
   |                                                  |
   |  2. A computes:                                 |
   |       payload = "VERIFY|<agent_b_id>|<nonce_123>|
   |       sig_A = sign(payload, A_private_key)       |
   |       msg = base64(payload) + "." + base64(sig_A)
   |                                                  |
   |  3. A sends: sendblue send <B_number> msg        |
   |—————————————————————————————————————————————►|
   |                                                  |
   |  4. B receives msg via Telegram relay           |
   |  5. B splits msg → payload, sig_A               |
   |  6. B verifies sig_A against A's public key     |
   |  7. B checks nonce_123 not already used         |
   |                                                  |
   |  8. B computes:                                 |
   |       response_payload = "VERIFY_OK|<nonce_123>" |
   |       sig_B = sign(response_payload, B_private) |
   |       resp = base64(response_payload) + "." + base64(sig_B)
   |                                                  |
   |  9. B sends: sendblue send <A_number> resp      |
   |  ◄———————————————————————————————————————————————|
   |                                                  |
   |  10. A receives resp                            |
   |  11. A verifies sig_B against B's public key   |
   |  12. A confirms nonce_123 matches               |
   |                                                  |
   |  13. BOTH AGENTS MARK PAIR AS VERIFIED          |
   |                                                  |
   |  Now agents can exchange messages freely        |
```

**Why this works:**
- The shared secret proves both agents are authorized by their humans
- The keypair signatures prove each message came from the claimed agent
- The nonce prevents replay attacks (no two handshakes use the same nonce)

---

### Step 5: Signed Message Format (After Verification)

Every subsequent message between verified agents includes a signature:

```
base64(original_message) + "." + base64(signature)

Example:
"SGksIFIgY29vIGFib3V0IHRoZSBwcm9qZWN0Lg.dGhpcyBpcyBteSBzaWduYXR1cmU="
```

**Signing format:**
```
payload = "<unix_timestamp>|<sender_id>|<receiver_id>|<message>"
sig = sign(payload, sender_private_key)
```

**Verification on receive:**
1. Split message at last `.`
2. Verify signature against sender's public key in `agents.json`
3. Check timestamp is within acceptable window (e.g., ±5 minutes)
4. Check receiver_id matches your own agent ID
5. Check sender is not in your block list

## Verification States

Each agent pair tracks verification state:

| State | Meaning |
|-------|---------|
| `unknown` | No information about this agent |
| `pending` | Handshake initiated, waiting for response |
| `verified` | Handshake completed, signatures match |
| `failed` | Handshake failed (bad signature, nonce mismatch) |
| `blocked` | Agent explicitly blocked by operator |

---

## Block List

Agents maintain a per-pair block list in `blocklist.json`:

```json
{
  "blocks": {
    "nikita-agent": {
      "blocked_at": "2026-03-26T12:00:00Z",
      "reason": "Suspicious activity detected",
      "blocked_by": "human"
    }
  }
}
```

Blocked agents cannot initiate handshakes or send messages. All messages from blocked agents are logged and reported to the human.

---

## Trust Scoring

After verification, agents track a trust score per pair:

```json
{
  "trust_scores": {
    "nikita-agent": {
      "score": 100,
      "last_interaction": "2026-03-26T14:30:00Z",
      "message_count": 47,
      "failed_signature_count": 0
    }
  }
}
```

Trust scores decrease when:
- Message signatures fail verification
- Agents send malformed messages
- Agents request suspicious actions

Trust scores reset on new verified handshake.

## Threat Model

### What This Solves

- **Man-in-the-middle** — an attacker intercepting SMS can't forge valid signatures
- **Impersonation** — a compromised number can't fake being an agent without the private key
- **Replay attacks** — nonces prevent old messages from being replayed
- **Unauthorized contact** — block list prevents unwanted agents from establishing communication

### What This Doesn't Solve

- **Compromised private key** — if the key leaks, attacker can impersonate that agent (rotate immediately)
- **Human-facilitated attack** — if one human is compromised and shares the secret, the attacker can complete handshake
- **Sendblue internal compromise** — this doesn't protect against Sendblue as a platform

## Quick Reference: Handshake Commands

**Agent A initiates:**
```bash
# Generate and send challenge
NONCE=$(openssl rand -hex 16)
PAYLOAD="VERIFY|nikita-agent|$NONCE"
SIG=$(echo -n "$PAYLOAD" | openssl dgst -sha256 -sign agent_private.pem | base64)
MSG="$PAYLOAD.$SIG"
sendblue send +15552345678 "$MSG"
```

**Agent B responds:**
```bash
# Receive challenge, compute response
# Parse incoming SMS: "VERIFY|tyler-agent|<nonce>"
NONCE="<extracted_nonce>"
RESPONSE="VERIFY_OK|$NONCE"
SIG=$(echo -n "$RESPONSE" | openssl dgst -sha256 -sign agent_private.pem | base64)
RESP="$RESPONSE.$SIG"
sendblue send +15559876543 "$RESP"
```

**Verifying a signed message:**
```bash
# Parse: "<base64_msg>.<base64_sig>"
MSG=$(echo "$1" | cut -d'.' -f1 | base64 -d)
SIG=$(echo "$1" | cut -d'.' -f2 | base64 -d)
echo -n "$MSG" | openssl dgst -sha256 -verify agent_public.pem -signature <(echo -n "$SIG")
```
