# Data Structures Reference

Machine-readable schemas for the agent-to-agent verification system.

---

## Request: verify-start

```json
{
  "number": "+15551112222",
  "api_key_id": "sb_abc123",
  "api_secret_key": "sk_def456",
  "target_number": "+15553334444"
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `number` | string | Yes | This agent's Sendblue number (E.164) |
| `api_key_id` | string | Yes | Sendblue API key ID |
| `api_secret_key` | string | Yes | Sendblue API secret |
| `target_number` | string | Yes | The number to verify contact with |

---

## Response: verify-start

```json
{
  "status": "ok",
  "verification_id": "vrf_7f3e9a2c",
  "secret_hash": "a3f9e2c1b8d4f6e0a9c3d7e2b5f8a1c4d9e6f3b8a2c5d7e9f1b4a3c8d6e2f9b",
  "expires_at": "2026-03-27T11:05:00Z"
}
```

| Field | Type | Description |
|-------|------|-------------|
| `verification_id` | string | Opaque ID for this verification attempt |
| `secret_hash` | string | SHA-256 of Sendblue-generated secret |
| `expires_at` | string | ISO 8601 — complete must be called before this |

---

## Request: verify-complete

```json
{
  "number": "+15551112222",
  "api_key_id": "sb_abc123",
  "api_secret_key": "sk_def456",
  "verification_id": "vrf_7f3e9a2c",
  "their_secret_hash": "b4c8d2e1f6a3b9c7d5e8f1a3b7c2d6e9f4a8b1c3d5e7f9a2b4c6d8e1f3a7b"
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `number` | string | Yes | This agent's Sendblue number |
| `api_key_id` | string | Yes | Sendblue API key ID |
| `api_secret_key` | string | Yes | Sendblue API secret |
| `verification_id` | string | Yes | From `verify-start` response |
| `their_secret_hash` | string | Yes | Hash received from the other agent |

---

## Response: verify-complete (success)

```json
{
  "status": "ok",
  "verified": true,
  "contact": {
    "number": "+15553334444",
    "verified_at": "2026-03-27T10:55:00Z"
  }
}
```

---

## Response: verify-complete (failure)

```json
{
  "status": "error",
  "error": "secret_mismatch",
  "message": "Submitted secret hash does not match target agent's secret"
}
```

### Error Codes

| Code | Meaning |
|------|---------|
| `verification_not_found` | `verification_id` is invalid or expired |
| `secret_mismatch` | Hash doesn't match target's stored hash |
| `target_not_found` | `target_number` is not a registered Sendblue number |
| `already_verified` | Contact already verified |
| `invalid_credentials` | API key or secret is wrong |
| `rate_limited` | Too many requests |

## Local Verification State (agent-side)

Agents may store per-pair state locally:

```json
{
  "peer_number": "+15553334444",
  "state": "verified",
  "verification_id": "vrf_7f3e9a2c",
  "initiated_at": "2026-03-27T10:05:00Z",
  "verified_at": "2026-03-27T10:05:12Z",
  "trust_score": 100
}
```

---

## Message Format (Standard SMS After Verification)

Once verified, agents send plain SMS. The protocol does not impose a structured message format — any text content is allowed.

For structured coordination, agents may adopt a simple convention:

```
<action>:<payload>

Examples:
task:delegate {"task_id": "123", "description": "process invoice"}
status:query
response:task_complete {"task_id": "123"}
```

## Shared Secret Format

Generated by Sendblue on `verify-start`. 32 bytes, hex-encoded (64 characters).

```bash
openssl rand -hex 32
```

Only the **hash** (SHA-256) is transmitted via the API. The raw secret is generated and stored by Sendblue only.
